The UR ROS 2 driver supports both direct joint-level torque control and the robot's built-in Cartesian force mode. These controllers let you implement compliant motion, surface probing, manual guidance, and other force-sensitive applications. This page covers the force mode controller, joint torque streaming, freedrive mode, tool contact detection, and the force/torque sensor broadcaster.

<Note>
For trajectory and position-based control, see [Position and velocity controllers](/controllers/position-velocity).
</Note>

## Force mode controller

### `force_mode_controller`

**Type:** `ur_controllers/ForceModeController`

The force mode controller activates the robot's built-in Cartesian force control, implemented via the URScript `force_mode(...)` function. Instead of commanding positions, you specify forces and torques relative to a task frame along with a selection vector that determines which Cartesian axes are force-controlled (compliant) and which remain position-controlled.

In compliant axes, the robot adjusts its position to achieve the specified force or torque. In non-compliant axes, the active position controller's motion commands execute normally.

<Warning>
This is not an admittance controller. Force constraints in a compliant axis override motion commands in that axis entirely. For example, specifying a force along the base frame's Z axis means any position-controller motion along Z will not be executed.
</Warning>

Force mode can be combined with all position control modes — `joint_trajectory_controller`, `passthrough_trajectory_controller`, `forward_position_controller`, `forward_velocity_controller`, and `motion_primitive_forward_controller` — to execute trajectories under force constraints.

<Tip>
The robot's force/torque sensor is a relative sensor. Zero it using the `~/zero_ftsensor` service on the `io_and_status_controller` before activating force mode and before the tool makes contact with the environment, for best accuracy.
</Tip>

### Activating and deactivating force mode

The controller is spawned inactive by default. Activate it first (alongside your chosen motion controller):

```bash
ros2 control switch_controllers --activate force_mode_controller
```

The controller exposes two services:

- `~/start_force_mode [ur_msgs/srv/SetForceMode]` — start force mode with specified parameters
- `~/stop_force_mode [std_srvs/srv/Trigger]` — stop force mode and return to normal position control

### Force mode parameters (`ur_msgs/srv/SetForceMode`)

<AccordionGroup>
  <Accordion title="task_frame">
    A `geometry_msgs/PoseStamped` defining the frame relative to which the selection vector, wrench, and limits are interpreted. The `frame_id` can be any frame transformable to the robot's `base` frame.
  </Accordion>
  <Accordion title="selection_vector_x/y/z/rx/ry/rz">
    Boolean flags (one per Cartesian axis of the task frame). Set to `1` to make the robot compliant (force-controlled) along that axis. Set to `0` to keep that axis position-controlled.
  </Accordion>
  <Accordion title="wrench">
    A `geometry_msgs/Wrench` specifying the forces (N) and torques (Nm) the robot will apply to its environment. Values have no effect on non-compliant axes. The actual wrench applied may be lower than requested due to joint safety limits.
  </Accordion>
  <Accordion title="type">
    An integer `[1, 3]` specifying how the force frame is interpreted:

    | Value | Behavior |
    |---|---|
    | `1` | The force frame's Y axis is aligned with the vector from the robot TCP towards the origin of the force frame |
    | `2` | The force frame is not transformed |
    | `3` | The force frame's X axis is the projection of the TCP velocity vector onto the X-Y plane of the force frame |
  </Accordion>
  <Accordion title="speed_limits">
    Maximum allowed TCP speed (m/s and rad/s) relative to the task frame. Only applies to axes marked as compliant in the selection vector.
  </Accordion>
  <Accordion title="deviation_limits">
    For non-compliant axes: the maximum allowed deviation (m and rad) between the actual TCP position and the position commanded by the active motion controller.
  </Accordion>
  <Accordion title="damping_factor">
    Force mode damping, in the range `[0, 1]`. Default is `0.025`. A value of `1` means full damping (robot decelerates quickly when no force is applied). A value of `0` means no damping (robot maintains speed).
  </Accordion>
  <Accordion title="gain_scaling">
    Force mode gain scaling, in the range `[0, 2]`. Default is `0.5`. Values above `1` can destabilize force mode, especially when pushing against hard surfaces or during collisions.
  </Accordion>
</AccordionGroup>

### Controller parameters

<Accordion title="force_mode_controller parameters">

| Parameter | Type | Default | Description |
|---|---|---|---|
| `tf_prefix` | string | `""` | URDF prefix of the corresponding arm |
| `check_io_successful_retries` | int | `10` | Number of retries when verifying that force mode was activated successfully |

</Accordion>

### Example

An example demonstrating force mode is available at `ur_robot_driver/examples/force_mode.py`. It shows how to:

<Steps>
  <Step title="Switch to the passthrough and force mode controllers">
    ```bash
    ros2 control switch_controllers \
      --deactivate scaled_joint_trajectory_controller \
      --activate passthrough_trajectory_controller force_mode_controller
    ```
  </Step>
  <Step title="Move the robot to a starting pose">
    Send a `FollowJointTrajectory` goal via the `passthrough_trajectory_controller`.
  </Step>
  <Step title="Start force mode">
    Call `~/start_force_mode` with your desired task frame, selection vector, wrench, and limits.
  </Step>
  <Step title="Execute constrained motion">
    The robot executes motion from the active trajectory controller while applying the specified force constraints on compliant axes.
  </Step>
  <Step title="Stop force mode">
    Call `~/stop_force_mode` when the constrained motion is complete.
  </Step>
</Steps>

## Joint torque control

### `forward_effort_controller`

**Type:** `forward_command_controller/ForwardCommandController` with `interface_name: effort`

Streams target joint torques directly to the robot via the URScript `direct_torque(...)` function. The robot automatically compensates for gravity, so the commanded torques should not include gravity compensation. The controller subscribes to `~/commands [std_msgs/msg/Float64MultiArray]` with one torque value (Nm) per joint.

<Note>
This controller requires PolyScope 5.25.1 or later (PolyScope 5 series), or PolyScope 10.12.1 or later (PolyScope X series).
</Note>

<Warning>
The `effort` field in `sensor_msgs/JointState` (published by `joint_state_broadcaster`) contains motor currents, not physical joint torques. Do not use it as torque feedback.
</Warning>

To activate:

```bash
ros2 control switch_controllers \
  --deactivate scaled_joint_trajectory_controller \
  --activate forward_effort_controller
```

The `forward_effort_controller` is mutually exclusive with all other joint-level motion controllers (position, velocity, and trajectory).

### Friction compensation — `friction_model_controller`

**Type:** `ur_controllers/FrictionModelController`

When using direct torque control, the robot applies internal friction compensation to the joints. The `friction_model_controller` lets you tune per-joint viscous and Coulomb friction scale factors that control how much compensation is applied. Each scale factor is in `[0, 1]`, where `0` means no compensation and `1` means full compensation.

The controller is **active by default** and exposes one service:

- `~/set_friction_model_parameters [ur_msgs/srv/SetFrictionModelParameters]` — set per-joint viscous and Coulomb friction scale factors. Each array must contain exactly one value per joint.

<Note>
Adjusting friction parameters only has an observable effect during direct torque control. The controller is technically compatible with all motion controllers and can remain active at all times.
</Note>

## Freedrive mode controller

### `freedrive_mode_controller`

**Type:** `ur_controllers/FreedriveModeController`

Freedrive mode lets you manually move the robot's joints without any active control from ROS. This is useful for teaching poses, physically guiding the robot through a task, or simply repositioning the arm by hand.

The controller cannot be combined with any other motion controller. Activate it by deactivating the current motion controller first:

```bash
ros2 control switch_controllers \
  --deactivate scaled_joint_trajectory_controller \
  --activate freedrive_mode_controller
```

### Enabling and disabling freedrive

Publish `True` messages continuously on `~/enable_freedrive_mode [std_msgs/msg/Bool]` to keep freedrive active. If no message is received within the `inactive_timeout` window (default: 1 second), freedrive is automatically deactivated.

```bash
ros2 topic pub --rate 2 /freedrive_mode_controller/enable_freedrive_mode \
  std_msgs/msg/Bool "{data: true}"
```

To deactivate, publish `False`, stop publishing, or deactivate the controller:

<CodeGroup>

```bash Publish False
ros2 topic pub --once /freedrive_mode_controller/enable_freedrive_mode \
  std_msgs/msg/Bool "{data: false}"
```

```bash Deactivate controller
ros2 control switch_controllers \
  --deactivate freedrive_mode_controller \
  --activate scaled_joint_trajectory_controller
```

</CodeGroup>

<Tip>
Publish at least twice the inverse of `inactive_timeout` per second to keep freedrive reliably active. With the default 1-second timeout, publish at 2 Hz or faster.
</Tip>

### Parameters

| Parameter | Type | Default | Description |
|---|---|---|---|
| `tf_prefix` | string | `""` | URDF prefix of the corresponding arm |
| `inactive_timeout` | int | `1` | Seconds of message inactivity after which freedrive is automatically deactivated |

## Tool contact controller

### `tool_contact_controller`

**Type:** `ur_controllers/ToolContactController`

When active, this controller enables the robot's tool contact detection. The robot monitors for tool contact in the current TCP direction of movement. When contact is detected, the robot stops all motion and retracts to the position where contact was first sensed.

This is useful for probing, surface detection, and safe approach applications.

The controller can run alongside `joint_trajectory_controller` or `passthrough_trajectory_controller`.

### Usage

Activate the controller:

```bash
ros2 control switch_controllers --activate tool_contact_controller
```

Send an action goal to begin monitoring for contact:

```bash
ros2 action send_goal /tool_contact_controller/detect_tool_contact \
  ur_msgs/action/ToolContact
```

The action has no fields — sending the goal is equivalent to enabling tool contact detection. The action succeeds when contact is detected, and remains active until contact is detected, cancelled by the user, or aborted by the hardware.

### Parameters

| Parameter | Type | Default | Description |
|---|---|---|---|
| `tf_prefix` | string | `""` | URDF prefix of the corresponding arm |
| `action_monitor_rate` | double | `20.0` | Rate (Hz) at which the action server monitors for contact |

## Force/torque sensor broadcaster

### `force_torque_sensor_broadcaster`

**Type:** `force_torque_sensor_broadcaster/ForceTorqueSensorBroadcaster`

Publishes the TCP wrench as reported by the robot's built-in force/torque sensor. This broadcaster is read-only and runs alongside any other controller.

| Topic | Type | Description |
|---|---|---|
| `ft_data` | `geometry_msgs/msg/WrenchStamped` | TCP wrench (forces in N, torques in Nm) in the `tool0_controller` frame |

The sensor measures relative changes in force and torque, not absolute values. Zero it before use with:

```bash
ros2 service call /io_and_status_controller/zero_ftsensor std_srvs/srv/Trigger
```

## Controller compatibility summary

| Controller | Can combine with | Mutually exclusive with |
|---|---|---|
| `forward_effort_controller` | `friction_model_controller`, `force_torque_sensor_broadcaster` | All other joint-level motion controllers |
| `force_mode_controller` | All position controllers (`joint_trajectory_controller`, `passthrough_trajectory_controller`, `forward_position_controller`, etc.) | None (supplementary) |
| `friction_model_controller` | All controllers | None (only affects torque control behavior) |
| `force_torque_sensor_broadcaster` | All controllers | None (read-only) |
| `freedrive_mode_controller` | `force_torque_sensor_broadcaster` | All other motion controllers |
| `tool_contact_controller` | `joint_trajectory_controller`, `passthrough_trajectory_controller`, `motion_primitive_forward_controller` | Streaming controllers |
