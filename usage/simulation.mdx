The ROS2 driver supports two simulation approaches: the official URSim simulator, which is functionally equivalent to a real robot from the driver's perspective, and ros2_control's mock hardware, which mirrors commands to states without any robot kinematics. Both let you develop, test, and validate launch files and controller configurations before connecting to physical hardware.

## URSim

Universal Robots provides Docker images of the URSim simulator for each PolyScope generation. The driver treats URSim identically to a real robot.

### Starting URSim

The `ur_client_library` package ships a helper script that provides a unified startup interface regardless of PolyScope version:

```bash
ros2 run ur_client_library start_ursim.sh -m <ur_type> -v <ursim_version>
```

- `-m` sets the robot model (defaults to UR5e if omitted).
- `-v` sets the PolyScope version (defaults to the latest PolyScope 5 image if omitted).

**Example** — start a UR20 with PolyScope X 10.8.0:

```bash
ros2 run ur_client_library start_ursim.sh -v 10.8.0 -m ur20
```

Expected output:

```

ROBOT_SERIES: polyscopex

        http://192.168.56.101

```

The script also installs the External Control URCap automatically.

### Connecting the driver to URSim

Once URSim is running, launch the driver pointing at the simulator's IP address (the default is `192.168.56.101`):

```bash
ros2 launch ur_robot_driver ur_control.launch.py \
  ur_type:=ur20 \
  robot_ip:=192.168.56.101 \
  launch_rviz:=true
```

After the driver starts, follow the same [startup steps](/usage/startup) as with a real robot — load the External Control program in the simulator and press **Play** (or use headless mode). When you move the robot in PolyScope, the RViz visualization updates accordingly.

## Mock hardware

For testing launch files, controller configurations, and URDF descriptions without any robot or simulator, use ros2_control's `MockSystem` by passing `use_mock_hardware:=true`:

```bash
ros2 launch ur_robot_driver ur_control.launch.py \
  ur_type:=ur5e \
  robot_ip:=yyy.yyy.yyy.yyy \
  use_mock_hardware:=true
```

The mock system mirrors every command directly to the joint states, so controllers and ROS interfaces behave as expected — but there is no real kinematics or physics.

<Note>
  Several driver features do not work with mock hardware:

  - The TCP pose broadcaster is disabled.
  - The passthrough trajectory controller does not execute `follow_joint_trajectory` actions.
  - The force mode controller does not respond to force mode requests.
  - The GPIO controller cannot verify I/O pin state changes and will report a failure when attempting to set a pin.
</Note>
